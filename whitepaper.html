Section 01

Abstract

The Omen protocol provides a decentralized reputation layer for the Sui ecosystem. By linking human founders to their autonomous agents via zkLogin-verified OmenBadges, we establish a permanent accountability layer that operates natively within Move Programmable Transaction Blocks.

The system utilizes Walrus for secure storage of large audit metadata and Move PTBs for on-chain enforcement. Unlike traditional oracle-based trust systems, Omen operates as a PTB firewall — returning routing decisions rather than blocking transactions, ensuring full compatibility with existing DEX aggregators.

Core Thesis

"Web3 verifies smart contracts, but not who built them. Omen blocks malicious transactions before they execute — on-chain, at the trade level, native to Sui."

The Trust Gap

Web3 lacks a non-transferable reputation layer. A beautiful interface can hide a malicious team. Omen fills this by linking real-world builder credibility to soulbound on-chain identities, verified before capital moves.

The Agentic Economy Problem

As AI agents take over trading and governance, anonymous agents present an existential risk to on-chain capital. Omen provides the Proof-of-Creator layer needed for autonomous safety — cryptographically binding every agent to its human founder.

Section 02

Identity Primitive

The foundation of the protocol is the OmenBadge — a soulbound Move object that cannot be transferred, sold, or delegated. Reputation is not a commodity.

Design Principle

"By making the identity object immobile at the Move bytecode level, we ensure trust belongs to the builder, not the highest bidder."

Sybil Resistance — 50 USDC Staked Deposit

Badge minting is free in terms of gas, but requires a refundable 50 USDC staked deposit held by the protocol. This deposit is burned upon any successful slash event. This mechanism ensures that creating a new identity after a slash is economically costly, preventing reputation laundering through fresh wallets.

Move · OmenBadge init
public entry fun init_badge( ctx: &mut TxContext, deposit: Coin<USDC>, // 50 USDC required zk_proof: vector<u8>, // zkLogin proof ) { assert!(coin::value(&deposit) == 50_000_000, E_INSUFFICIENT_DEPOSIT); let badge = OmenBadge { id: object::new(ctx), owner: tx_context::sender(ctx), score: 50, // base score on mint deposit_locked: true, social_recovery_guardians: vector[], }; transfer::transfer(badge, tx_context::sender(ctx)); }

Social Recovery — 7-Day Timelock

Institutional users can register up to 5 guardians for badge recovery. A 3-of-5 guardian threshold with a 7-day timelock allows teams to recover badge access after key loss or team transitions — without forfeiting reputation history accumulated over time.

Section 03

Seek & Remit Architecture

The protocol operates as a read/write primitive with two distinct interaction modes.

Seek — Reading Trust

Seek refers to wallets, dApps, and DEX aggregators querying the OmenRegistry in real-time. Any party can call OmenRegistry::get_score to retrieve a builder's current trust score before approving an interaction.

Remit — Writing Audit Data

Remit refers to credentialed auditors pushing verified audit results back into the builder's score. Unlike permissionless vulnerability submission models that create audit cartels, only approved AuditorBadge holders can write to the registry. See Section 06 for the full AuditorBadge approval process.

Section 04

Storage Layer

Heavy metadata — raw JSON outputs from AI code audits, full audit reports, agent behavior logs — is stored via Walrus Protocol. OmenBadge objects store a 256-bit hash and a Blob ID using Sui Dynamic Fields, ensuring O(1) verification efficiency without bloating on-chain state.

Why Walrus

Security data must outlive projects. A builder's audit history remains verifiable on Walrus even if the project disbands or the team disappears. The cryptographic anchor on Sui ensures the data cannot be silently altered.

Section 05

Execution & Routing

Omen integrates with DeepBook V3 to gate trading pools. The core mechanism is a PTB firewall router — not an oracle and not a hard revert. This distinction is critical for DEX aggregator compatibility.

assert_trade_safety — Routing Skip, Not Revert

The assert_trade_safety function returns (bool, u64). A false return is a routing skip — the DEX routes the trade to an alternative pool. No transaction fails. No user sees an error. The risk is silently routed around.

This is what separates Omen from any system using assert!. A hard revert breaks aggregator compatibility. A routing skip is invisible to the user and non-breaking to the integration.

Move · PTB Router
public fun assert_trade_safety( registry: &OmenRegistry, builder_badge: address, min_score: u64, ): (bool, u64) { // (safe_to_route, rejection_code) let score = get_score(registry, builder_badge); if (score >= min_score) { (true, 0) // route normally } else { (false, E_SCORE_BELOW_THRESHOLD) // routing skip } }

DeepBook V3 Pool Gating

Protocols can enforce a minimum trust score before allowing a builder to create or manage a liquidity pool, preventing initial seed attacks and anonymous capital injection from unverified actors.

Section 06

AuditorBadge System

Omen explicitly rejects the open vulnerability submission model. Permissionless audit submissions create audit cartels — coordinated groups that inflate or deflate scores for competitive advantage.

3-of-3 Sub-Committee Approval

All AuditorBadge approvals require unanimous sign-off from a dedicated three-member sub-committee:

SeatRoleRationale
Seat 1Sui Foundation representativeEcosystem alignment
Seat 2Independent Move security researcherTechnical independence
Seat 3Existing tier-1 auditorPeer accountability

Current approved auditors include firms operating at the tier-1 level of the Move security ecosystem (MoveBit, Zellic, OtterSec class). All three seats must approve before an AuditorBadge is minted.

Design Principle

"Only credentialed auditors write to the registry. This is not decentralisation theatre — it is institutional-grade accountability with a permissioned write layer and a permissionless read layer."

Section 07

Agent Lineage Graph

AI child agents are cryptographically linked to their creators via the Lineage::verify_parent primitive. The accountability is inherited and the slash cascade is hardcoded — not governance-controlled.

Cascade Slash Percentages

LevelSlash AmountNotes
Level 1 (direct child)100% agent stake + 25% parent stakeFull agent burn, partial parent penalty
Level 2 (grandchild)15% grandparent stakeCascade stops here for third-party agents with waiver
Level 3+No cascadeLineage accountability terminates at Level 2

Third-Party Agent Liability

Third-party agents (agents not created by the badge holder but operating on their behalf) require an explicit LIABILITY_WAIVER flag set on deployment. Without this flag, third-party agents are treated as direct Level 1 children for slash purposes. This prevents founders from offloading accountability to subcontracted agent builders.

Section 08

Resilience Infrastructure

SDK Fallback Architecture

The Omen SDK operates a 2-second hard timeout before automatic L1 RPC fallback. If Omen's backend infrastructure goes down, the system continues routing using direct on-chain reads. This is the institutional trust message: Omen's backend is a performance optimisation, not a dependency.

LayerResponse TimeFallback Behaviour
Redis cache (primary)~20msServes cached score
Omen API (secondary)~200msFresh on-chain read
L1 RPC (fallback)2s timeout triggerDirect Sui fullnode query

OmenRegistry Hybrid Indexing

The OmenRegistry uses a hybrid indexing model with a custom PostgreSQL stack and failover RPC polling to ensure high data availability for institutional B2B partners. The on-chain state is always the source of truth — the off-chain index is a read optimisation layer only.
